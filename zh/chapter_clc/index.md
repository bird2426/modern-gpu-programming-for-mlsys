:::{admonition} 概览
:class: overview

- 持久化内核保持固定数量的 CTA 或 CTA 集群常驻（通常大小设置为每个 SM 大致有一个活跃的工作拥有者，但不依赖有保障的一一映射），并让它们循环处理多个输出 tile，而不是为每个 tile 启动一个 CTA。
- Cluster Launch Control 是 Blackwell 的硬件机制，允许常驻集群在运行时请求另一个 tile。这是一个基于两条 PTX 指令构建的硬件任务窃取路径：一条指令请求任务，另一条读取请求是否成功。
- 主要好处是更好的尾部行为。当 tile 的代价不均匀，或者 tile 数量无法在可用 SM 之间平均分配时，提前完成的 CTA 可以拉取更多任务，而不是空闲等待。
:::

持久化 GEMM 不将 CUDA 网格视为固定的每个输出 tile 一个 CTA 的启动方式。相反，它启动较少数量的长生命周期 CTA 或 CTA 集群。每个 CTA 计算一个 tile，前进到下一个 tile，再次计算，直到输出空间全部完成。这就是在 {ref}`chap_gemm_advanced` 中建立的执行模式。

一旦内核是持久化的，主要的调度问题就变得简单了：在 CTA 或集群完成当前 tile 后，下一个 tile 从哪里来？

最简单的答案是静态公式。例如，内核可以从 CTA id 计算 tile 坐标，然后按网格步长前进。这很容易实现，并且当所有 tile 的成本大致相同且 tile 数量在 GPU 上均匀分布时效果很好。但调度是在工作实际运行之前决定的。如果某些 tile 花费更长时间，或者最后几个 tile 分配不均匀，一些 SM 提前完成了它们的份额，而其他 SM 仍在处理尾部。

Cluster Launch Control，简称 CLC，改变了这种调度模型。持久化集群不是预先决定整个分配，而是可以向硬件网格调度器请求另一个尚未启动的集群的任务。如果请求成功，当前集群接管该集群坐标并计算相应的 tile。如果请求失败，则没有更多任务可窃取，循环退出。

这与线程块集群本身不同。线程块集群（一起启动的 CTA，具有集群级同步和分布式共享内存访问）是在 Hopper 中引入的（{ref}`chap_background`）。CLC 是 Blackwell 的新增功能，使得在这些集群坐标上的调度变为动态。集群已经是启动的单位；CLC 允许已经运行的集群取消待启动并继承其坐标。

## 两条指令

Cluster Launch Control 通过两条 PTX 指令暴露。第一条指令向网格调度器发送异步请求。第二条指令读取响应。

请求指令是 `clusterlaunchcontrol.try_cancel.async`。

`try_cancel` 请求调度器取消待启动集群的启动并将该集群的坐标返回给调用者。响应以 16 字节记录写入共享内存。由于请求是异步的，该指令不会等待响应到达。相反，完成状态通过 `mbarrier` 报告，使用与 TMA 相同的 barrier 加 phase 模型。

这是一个重要的细节，因为这意味着 CLC 没有引入新的等待模型。内核发出请求，将其与 barrier 关联，然后等待 barrier 完成后再读取响应。响应的到达通过 barrier 以字节计数完成的方式发出信号，与其他异步硬件操作的一般风格相同（参见 {ref}`chap_async_barriers`）。

一旦 barrier 触发，内核就使用查询指令。

第一个查询是 `clusterlaunchcontrol.query_cancel.is_canceled`。它返回一个谓词，告诉内核取消操作是否成功。真谓词表示调度器找到了一个待启动的集群，取消了它，并返回了它的坐标。假谓词表示没有剩余的可获取的待处理任务。

只有当 `is_canceled` 为真时，内核才应该读取坐标。它使用 `clusterlaunchcontrol.query_cancel.get_first_ctaid` 来读取，该指令提取被取消集群的第一个 CTA id。该 CTA id 是一个坐标向量，通常读作 `(x, y, z)`，内核将其解码为下一个应该计算的输出 tile。

此协议中没有数值哨兵 tile id。内核根据谓词分支。如果谓词为真，则坐标有效。如果谓词为假，则任务窃取循环结束。

在底层，这种形态直接源于 CLC 的工作方式。硬件不是从软件队列中分配抽象任务。它是在取消一个尚未发生的集群启动。因此，成功的响应包含一个真实的集群坐标。失败的响应仅表示启动队列已耗尽。

## Work-Stealing Loop

有了这两条指令，持久化调度器就变成了一个短循环。

在循环中的任何时候，集群有一个它负责计算的 tile。在开始该 tile 之前，它为下一个 tile 发送 `try_cancel` 请求。该请求异步运行。调度器处理请求的同时，集群计算其当前 tile。

当前 tile 完成后，集群等待与 `try_cancel` 响应关联的 `mbarrier`。响应就绪后，它调用 `query_cancel.is_canceled`。如果谓词为真，它调用 `query_cancel.get_first_ctaid`，解码返回的坐标，并将其用作下一个 tile。如果谓词为假，则没有剩余任务，集群退出。

在代码形式上，循环是：

1. 为可能的下一个 tile 发起 `try_cancel`；
2. 在请求进行中计算当前 tile；
3. 等待响应 barrier；
4. 查询取消是否成功；
5. 继续使用返回的坐标或退出。

请求的放置位置是使循环有用的关键。集群不会等到完成当前 tile 后才请求更多任务。它先请求，再计算。这将调度器请求与有用的工作重叠。到当前 tile 完成时，下一个 tile 的答案通常已经可用。

这与持久化内核在其他地方使用异步拷贝和 tensor-core barrier 的基本原因相同。内核避免将长延迟操作直接放在关键路径上。CLC 将相同的思路应用于 tile 调度：尽早请求下一个工作单元，计算当前单元，然后在需要时消耗调度结果。

## 与 Persistent GEMM 的关系

{ref}`chap_gemm_advanced` 中的持久化 GEMM 在主流程讲解中使用静态调度器。静态调度器更容易解释，因为下一个 tile 可以直接从循环状态计算。例如，`ClusterPersistentScheduler2D` 这样的调度器可以使用网格步长模式在输出 tile 空间上分配 tile。

CLC 是该静态分配的动态替代方案。外层循环保持不变：每个常驻集群重复计算一个输出 tile，然后前进到下一个。变化的是下一个 tile 的来源。使用静态调度器时，下一个 tile 通过公式计算。使用 CLC 时，下一个 tile 通过硬件任务窃取返回。

这种差异在启动的尾部附近最为重要。在静态调度中，剩余的工作可能无法均匀分布。一些 SM 可能已经用完分配的 tile，而其他 SM 还有几个剩余。使用 CLC 时，提前完成的集群请求另一个待启动的集群坐标。只要启动队列中还有任务，提前完成的集群就会持续拉取更多 tile。

当 tile 代价不均衡时，这也同样重要。某些 GEMM tile 可能因为边界、掩码、稀疏性、分组调度或主矩阵乘法周围的融合操作而走不同的路径。静态调度假设在观察到这些代价之前 tile 分配已经足够好。CLC 不需要这个假设。它只在集群可用后才分配更多任务。

因此，在 TIRx 中，CLC 可以暴露为动态 tile 调度器。编程模型不需要改变 tile 的计算方式。tile 主体与静态调度器使用的持久化 GEMM 主体相同。调度器从"根据公式计算下一个 tile 坐标"变为"向硬件请求下一个可用集群坐标"。结果是相同的持久化循环，但使用硬件驱动的任务分配，而非固定的启动时调度。
